# 用户故事模板和指南

## 标准格式

```
As a [用户类型]
I want [功能/能力]
So that [价值/收益]
```

## INVEST 检查清单

- [ ] **I**ndependent (独立) - 故事可以独立开发，不依赖其他故事
- [ ] **N**egotiable (可协商) - 细节可以讨论调整
- [ ] **V**aluable (有价值) - 为用户或业务提供价值
- [ ] **E**stimable (可估算) - 团队可以估算工作量
- [ ] **S**mall (小) - 可在一个迭代内完成
- [ ] **T**estable (可测试) - 有明确的验收标准

## 示例对比

### 好的用户故事

```
As a 注册客户
I want 将商品保存到愿望清单
So that 我可以稍后购买而无需重新搜索
```

特点：

- 明确的用户角色
- 具体的功能描述
- 清晰的业务价值

### 差的用户故事

```
As a 用户
I want 系统更好用
So that 体验更好
```

问题：

- 用户角色太模糊
- 功能描述不具体
- 价值陈述空洞

## 验收标准格式

### Given-When-Then 格式

```
Given [前置条件/上下文]
When [用户操作/触发事件]
Then [预期结果]
```

### 检查清单格式

```
- [ ] 用户可以 [操作]
- [ ] 系统显示 [反馈]
- [ ] 数据被 [验证/存储/处理]
```

## 故事拆分策略

当故事过大时，按以下维度拆分：

### 1. 按工作流步骤

```
原始: 用户完成购物流程
拆分:
- 用户添加商品到购物车
- 用户查看购物车
- 用户填写收货信息
- 用户完成支付
```

### 2. 按业务规则

```
原始: 用户注册账号
拆分:
- 用户使用邮箱注册 (基础流程)
- 系统验证邮箱唯一性
- 系统发送验证邮件
- 用户处理注册失败情况
```

### 3. 按数据变化

```
原始: 用户管理商品
拆分:
- 用户添加单个商品
- 用户批量导入商品
- 用户编辑商品信息
- 用户删除商品
```

### 4. 按界面类型

```
原始: 用户访问控制面板
拆分:
- 用户在 Web 端访问控制面板
- 用户在移动端访问控制面板
- 管理员通过 API 访问数据
```

### 5. 按 CRUD 操作

```
原始: 用户管理联系人
拆分:
- Create: 用户创建新联系人
- Read: 用户查看联系人列表
- Update: 用户编辑联系人信息
- Delete: 用户删除联系人
```

## 用户角色常见类型

| 领域 | 常见角色 |
|------|---------|
| 电商 | 游客、注册用户、VIP会员、店铺管理员、客服 |
| SaaS | 免费用户、付费用户、团队管理员、超级管理员 |
| 内部系统 | 操作员、主管、部门管理员、IT管理员 |
| 移动应用 | 新用户、活跃用户、高级用户 |
