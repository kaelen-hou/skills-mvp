## 各部分编写指南

### 1. 概述

#### 1.1 背景

说明为什么要做这个功能，解决什么问题。

```markdown
**现状问题**：
- 用户目前需要 [描述痛点]
- 现有方案存在 [描述不足]

**市场机会**：
- [竞品分析或市场趋势]
```

#### 1.2 目标

明确、可衡量的目标。

```markdown
**业务目标**：
- 提升用户注册转化率 X%
- 降低用户流失率 X%

**用户目标**：
- 用户可以在 X 分钟内完成 [任务]
```

#### 1.3 成功指标

定义如何衡量功能是否成功。

```markdown
| 指标 | 当前值 | 目标值 | 衡量方式 |
|------|--------|--------|----------|
| 注册转化率 | 30% | 45% | 埋点统计 |
| 任务完成时间 | 5分钟 | 2分钟 | 用户行为分析 |
```

### 2. 用户分析

#### 2.1 目标用户

定义用户画像。

```markdown
**主要用户**：
- 角色：[用户类型]
- 特征：[年龄/职业/技术水平]
- 使用场景：[何时何地使用]

**次要用户**：
- [其他相关用户群体]
```

#### 2.2 用户痛点

```markdown
| 痛点 | 严重程度 | 当前解决方案 |
|------|----------|--------------|
| [痛点1] | 高 | [现状] |
| [痛点2] | 中 | [现状] |
```

#### 2.3 用户场景

典型使用场景描述。

```markdown
**场景1：[场景名称]**
- 触发条件：用户想要 [目的]
- 操作步骤：1. xxx 2. xxx 3. xxx
- 期望结果：[用户获得什么]
```

### 3. 功能需求

#### 3.1 功能列表

```markdown
| 功能 | 优先级 | 描述 | 依赖 |
|------|--------|------|------|
| 功能A | P0 | [简述] | - |
| 功能B | P1 | [简述] | 功能A |
```

#### 3.2 功能详情

每个功能的详细说明。

```markdown
### 功能：[功能名称]

**功能描述**：
[一句话描述功能做什么]

**前置条件**：
- [用户需要满足什么条件]

**业务规则**：
1. [规则1]
2. [规则2]

**异常处理**：
| 异常情况 | 处理方式 |
|----------|----------|
| [情况1] | [处理] |
```

#### 3.3 用户故事

使用标准用户故事格式。

```markdown
**US-001: [故事名称]**

As a [用户角色]
I want [功能]
So that [价值]

**验收标准**：
- [ ] Given [条件], when [操作], then [结果]
```

### 4. 非功能需求

```markdown
**性能要求**：
- 页面加载时间 < 2秒
- API 响应时间 < 500ms
- 支持 1000 并发用户

**安全要求**：
- 敏感数据加密存储
- API 需要身份认证
- 防止 SQL 注入/XSS 攻击

**兼容性要求**：
- 浏览器：Chrome/Firefox/Safari 最新两个版本
- 移动端：iOS 12+, Android 8+
```

## PRD vs 用户故事 对比

| 维度 | PRD | 用户故事 |
|------|-----|----------|
| 详细程度 | 完整、全面 | 精简、聚焦 |
| 适用场景 | 新产品/大功能 | 迭代开发/小功能 |
| 读者 | 全团队 | 开发团队 |
| 包含内容 | 背景、分析、设计、计划 | 功能、验收标准 |
| 维护成本 | 高 | 低 |

## 何时使用 PRD

- 全新产品或模块
- 涉及多个团队协作
- 需要与利益相关者对齐
- 复杂功能需要完整文档

## 何时使用用户故事

- 敏捷迭代开发
- 功能范围明确
- 团队内部沟通
- 快速交付
